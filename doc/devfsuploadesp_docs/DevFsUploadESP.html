URL to upload-page

A URL to the ESP32/8266 host-address, port & path provides access to the upload-page. Depending on how Station-mode or AP-mode are coded then both modes may function.


<ESP-host-address[:port]><path>
<ESP-host-address[:port]>/upload ' /upload '    default path set-able by user code
e.g. 192.168.0.5/upload • Station-mode URL
       192.168.4.1:8011/upload • AP-mode URL usually has a different IP and explicit port

Upload Zone

The upload zone contains the basic functionality, which is uploading files, via Browser-UI operations.

upload dir: field


The upload dir: field is the ESP-target-directory path where files/directory will upload too. The field may be manually set (typing); or using a 'select' button within the 'List Dir(s)' listing.

NB: It must start & end with "/" to be valid.

WARNING: length of upload dir: field & file-name may not exceed FS specification. [Not checked.]

List Zone

The list zone allows viewing of files or directories kinds on the ESP's FS. Additionally, FS management buttons are provided to manipulate files/directories.

Only one list kind is viewable at a time (an ESP storage optimization) and is the same in presentation for SPIFFS and LittleFS.

NB: Directories have different implementations, but DevFsUploadESP has a consistent presentation for both.

Difference: SPIFFS - LittleFS

Implementation is different between the file-systems (FSs), Also, ESP32 and ESP8266 implementations differ.

DevFsUploadESP attempts to provide a Browser-UI that works for both FSs, due to compatible APIs.

Sketch Guild

Sketch code (optional usage/install)

  • The DevFsUploadESP library folders need to be "installed" in the IDE. These are only used if they are explicited #include coded in a sketch. Thus, the libraries may remain in the IDE library folders regardless on included or not.

    See: for IDE installation

  • DevFsUploadESP is designed to be optionally "installed" into an Arduino IDE sketch .ino file. The user may choose the use of SPIFFS or LittleFS to install: but not both. Two user actions are required in sketch code: (See: ddd.)

    IMPORTANT: (1) and (2) need to be in the same file as a #define is used to associate the two.

    1. "install" #include

      use libraries by including a FS-explicit ".h" library; either

             #include <DevFsUploadSPIFFS.h>      or     #include <DevFsUploadLittleFS.h>

      Optional: If the include line is missing or commented out, the library is "not installed".

      All dependent libraries will automatically be included.

    2. "setup" association

      in the sketch's 'setup' function section associating the 'DevFsUploadUpESP::handlers' into the ESP server structure require the following code:

         // install DevFsUpload server handle
#ifdef DevFsUploadInstall
   DevFsUploadInstall(server);
#endif
      		 OR 
         // install DevFsUpload server handle
#ifdef DevFsUploadInstall
   DevFsUploadInstall(&server); 
#endif

      This code may be left in place as DevFsUploadInstall is only defined if 'DevFsUploadSPIFFS.h' or 'DevFsUploadLittleFS.h' file is included in the compile.

Alternate DevFsUploadInstall

If a sketch-app uses '/upload' as a URL-path, an alternative URL-path for DevFsUploadESP's should be used. This is done via a 2nd parameter in the 'DevFsUploadInstall' call:

   // install DevFsUpload server handle
   #ifdef DevFsUploadInstall
       DevFsUploadInstall(server, "/myupload");   
       // or DevFsUploadInstall(&server, "/myupload"); 
   #endif

URL-path to the upload page would be: <ESP-host-address[:port]>/myupload

The 2nd parameter may be any appropriate string (NB: leading '/' is explicitly needed.)

      ------ server->onFileUpload(DevFsUploadESP::handleFileUpload);
      ------ server->on("/upload", HTTP_POST, DevFsUploadESP::handleOther);

About DevFsUploadESP

About DevFsUploadESP

DevFsUploadESP is an aid used for SPIFFS or LittleFS file-systems (FS), used with Arduino IDE and ESP32/ESP8266 modules (ESP). It provides a Browser-UI to manage the FS components.

Purpose

A development aid to manage files and/or directories on an ESP file-system so a user may perform FS actions
• without the need of an IDE,
• via a Browser-UI.

Usages

DevFsUploadESP allows a user, via a browser-UI, to network to an ESP and perform file-system (FS) actions:

  • upload file(s)/dir-structure to an ESP-target-directory
  • upload file(s)/dir-structure to manually defined ESP-target-directory
  • replace file on ESP after being editted
  • some basic file/directory management functionality like: delete, view, make-sub-directory, remove-directory,...
  • rapid edit and re-install HTML/CSS/JS files by file upload(replacement) (basically HTML development on the fly from a PC)

Background

The aid was developed to allow upload and basic management of files on an ESP's FS, which was initially SPIFFS. Since the 'DevFsUpload' library was published, an additional file system called LittleFS has been implemented on the ESP3266. With ESP32 likely in mid/late-2020.

DevFsUploadESP provides a 'FS-common-library' to be used with a 'SPIFFS-library' or 'LittleFS-library' arrangement on an ESP-device.

Supported Capability

Considerations


Defects vs design intent

DevFsUploadESP is a development aid and not a polished product. As such there may be nuisances for users which are design-intent. That is: is it necessary and/or would it impact ESP storage significantly.

Limitation

The ESP has finite storage which required design compromises: browser-UI pre-validation, ESP fault reporting, read-ability of the HTML/CSS/JS files provided to the browser,....

There are others compromises, one of which is 'accessiblity' in the browser-UI. Supporting full ease-of-accessibility would result in large amounts of storage on an ESP.
   • 'Tabs' between controls should work, but remain untested as each browser (FF, Chrome,...) works differently.
